# TASK-2026-03-31-004

## Goal

- 在已校准的权威口径下，分析为什么当前 full-train 的 best scalar `val_mIoU = 0.7316`、latest `val_mIoU = 0.7085` 低于 `support-only baseline = 74.6`，并据此决定下一轮应优先走调参重训、结构改动，还是补额外验证。

## Why now

- `TASK-2026-03-31-003` 已完成验证口径修正；当前 `TASK-2026-03-31-004` 对应 workflow smoke 为 `WARN`，且仅 `1` 个 stale（`handoff/latest_round.md` 未同步），不影响本轮对 full-train 的性能判断。

- 若不先分析 full-train 低于 baseline 的原因，就会把“可学习性不足”“多任务拖拽”“优化失衡”“过拟合”这些不同问题混成同一种实现需求。

## In scope

- 基于现有 full-train 日志、摘要和必要的 checkpoint / best-epoch 记录，分析 semantic scalar `val_mIoU` 低于 baseline 的原因。
- 判断 direction / axis-side 分支是否存在可学习性问题，或是否对 semantic 主任务形成拖拽。
- 分析 support 学到了什么，以及旧失败 run 中 `dist` 诊断值为何改善但不能被直接写成“当前 active rollout 中 dist 仍在学习”。
- 判断后期退化更像过拟合、优化失衡，还是多任务拖拽。
- 最后给出下一轮优先路线：调参重训、结构改动，还是补额外验证。

## Out of scope

- 不在本轮修改业务代码、模型主线、loss 主线、head、data pipeline 或 Pointcept。
- 不在本轮直接执行调参重训或结构改动。
- 不要求新训练，不追加新实验。
- 不扩展到 test/export/visualization 或跨多 task 历史整理。

- 已确认事实：当前 workflow smoke 的事实是 `WARN`，且仅 `1` 个 stale（`handoff/latest_round.md` 未同步），不影响本轮对 full-train 性能判断。

- 已确认事实：当前正式主线仍是 `axis + side + support`；旧 `support-shape` full-train 只作为失败旁证，不代表当前 active rollout 已被 full-train 直接验证。
- 已确认事实：旧失败 run 中方向类诊断值表现很差，`val_loss_dir` / `dir_cosine` 的较优点停在 epoch 1 一带；`support` 指标继续改善，而 `dist` 只体现为旧失败 run 中的诊断值改善，不是该轮主训练目标。
- 待解释问题：如何把旧失败旁证映射到当前 active `axis + side + support` 主线语言，并明确“哪一层耦合”最可能在拖拽 semantic `val_mIoU`。
- 当前最强假设：当前问题更像“方向类几何信号引入后的多任务耦合失衡”，而不是单纯调参失败；但这仍是最强假设，不是已被当前证据直接证明的事实。
- active 主线下已定点的真实耦合点：共享 backbone 特征同时供 semantic / boundary 分支使用；`SupportConditionedEdgeHead` 使 support 特征处于 axis/side 上游；`loss_semantic + loss_edge` 直接相加；trainer 对聚合后的单个 `loss` 做一次 backward / optimizer step。
- 当前唯一优先的 candidate principle：保留 support 对 boundary branch 的直接监督，但 axis/side 只能在 support-conditioned 私有分支上学习，其损失不再直接回传 shared backbone。
