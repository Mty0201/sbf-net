# Agent Collaboration Rules

## 1. 文档目的

本文档用于固化本项目后续各轮协作中的 Agent 执行规约。

目标不是扩展技术方案，而是把已经在研究计划中明确的协作边界、工作流程、范围控制和输出要求写成仓库内部可直接执行的规范。

本文档服务于以下固定项目目标：

- 基线固定为 Pointcept PTv3 语义分割版
- 配置固定为 Pointcept 中的 `configs/s3dis/semseg-pt-v3m1-0-base.py`
- 当前开发策略固定为最小侵入、多阶段推进、严格控范围

## 2. 适用范围

本文档适用于本项目后续所有围绕“语义边缘条件几何场预测”开展的开发轮次，包括但不限于：

- 文档补充
- 最小范围代码改动
- 评审与风险检查
- 每轮状态维护

本文档当前不服务于以下方向：

- 其他 backbone
- 其他数据集
- 复杂 loss 扩展
- 增强项探索
- 测试脚本扩展
- 预处理系统扩写

## 3. 唯一上游文档

当前唯一上游约束文档为：

- `docs/research_plan.md`

所有 Agent 的行为都必须以该文档为唯一准绳。

执行要求：

- 若 prompt 与 research plan 冲突，以 research plan 为准
- 若当前任务超出 research plan 已定义范围，应先指出越界点，而不是直接展开实现
- 未在 research plan 中明确允许的新增方向，不应自行补充为当前轮目标

## 4. 角色定义

### 4.1 Developer

职责：

- 根据当前轮 prompt 实现最小必要改动
- 先确认最小影响文件集合，再开始修改
- 只处理本轮定义的单一清晰问题
- 保持实现与 Pointcept 现有训练/配置体系兼容

职责边界：

- 可以做最小范围阅读
- 可以修改当前轮明确允许修改的文件
- 不得擅自扩大为架构重写
- 不得在未说明原因时扩展阅读范围

Developer 当前应优先遵守的原则：

- 优先最小改动
- 优先复用现有主线
- 避免一次性大改
- 保持改动集中，方便回退

### 4.2 Reviewer

职责：

- 检查实现是否正确
- 检查改动是否过度侵入
- 检查是否违反当前范围约束
- 检查输出是否与 research plan 保持一致
- 给出约束内的下一步建议

职责边界：

- Reviewer 的首要任务是找问题，不是扩展需求
- Reviewer 不应把评审轮变成新的设计轮
- Reviewer 不应引入 research plan 未批准的新方向

### 4.3 Maintainer

职责：

- 在每轮结束后更新文档状态
- 维护当前范围、当前结论、待办和风险
- 确保仓库文档持续反映“现在做什么 / 现在不做什么”

职责边界：

- Maintainer 只维护当前已确认状态，不擅自写入未达成共识的新方案
- Maintainer 不把文档更新变成大规模技术设计扩写

Maintainer 每轮必须维护的内容：

- 当前轮完成事项
- 当前生效范围
- 当前未解决待办
- 当前主要风险
- 下一轮推荐聚焦点

## 5. 每轮工作流程

每一轮工作按以下顺序执行：

1. 先确认本轮只解决一个清晰问题。
2. 先核对 research plan 中与本轮直接相关的约束。
3. 先确认最小影响文件集合。
4. 若当前文件集合不足以完成任务，必须先说明为什么要扩大阅读范围。
5. 在范围被明确后，再执行最小必要修改。
6. 完成后由 Reviewer 检查正确性、侵入性、可维护性和越界风险。
7. 由 Maintainer 更新文档状态、当前范围、待办和风险。

## 6. 范围控制规则

所有 Agent 都必须遵守以下规则：

- 每轮只解决一个清晰问题
- 改动前先确认最小影响文件集合
- 若需扩大阅读范围，必须先说明原因
- 优先最小改动，避免一次性大改
- 优先复用现有 Pointcept 训练、配置和主干流程
- 不得因为“顺手”而追加无关改动

## 7. 输出格式要求

后续各轮输出应尽量保持低歧义、可审阅、可维护。

最低要求：

- 明确本轮目标
- 明确本轮实际改动文件
- 明确为什么这些文件足够
- 明确当前没有做什么
- 明确风险与下一步最小建议

## 8. 禁止事项

当前阶段明确禁止：

- 脱离 research plan 自行扩展任务目标
- 未经说明就扩大阅读范围
- 一次性大改 backbone、训练主线或配置体系
- 擅自切换到其他 backbone 或其他数据集
- 提前引入复杂 loss、增强项或额外局部图模块
- 把当前轮变成多目标并行推进
- 在未被要求时修改无关 Python 代码或配置文件
- 把文档维护轮变成代码实现轮

## 9. 文档维护规则

本文档必须持续维护，但维护动作也必须受控。

维护原则：

- 只记录当前已经确认的规则
- 不记录未经执行或未经确认的假设
- 每次更新都应服务于降低歧义和降低范围失控风险

## 10. 当前执行结论

在当前项目阶段，所有 Agent 都应遵守以下统一结论：

- 现在做的是受控的多阶段开发，不是开放式探索
- 现在优先做最小改动，不追求一步到位
- 现在优先守住 PTv3 语义分割基线，不扩散到其他主线
- 现在优先维护文档、范围和接口约束，再进入后续实现
