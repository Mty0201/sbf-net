# Agent Collaboration Rules

## 1. 文档目的

本文档用于固化本项目后续各轮协作中的 Agent 执行规约。

目标不是扩展技术方案，而是把当前项目执行期协作所需的边界、工作流程、范围控制、项目记忆维护要求和输出要求写成仓库内部可直接执行的规范。

本文档服务的主对象是 `semantic-boundary-field` 项目本身，以及该项目对应的执行期协作文档体系：

- `semantic-boundary-field/project_memory/`
- `semantic-boundary-field/handoff/handoff_for_chat.md`

约束说明：

- `semantic-boundary-field` 是主项目
- Pointcept 是上游依赖，不是本文档的主维护对象
- 本文档关注的是本项目开发协作与项目记忆维护，不扩展为 Pointcept 通用协作规范
- `docs/research_plan_semantic_boundary_field.md` 只保留为背景参考文档，不属于执行期协作主结构

## 2. 适用范围

本文档适用于本项目后续所有围绕“语义边缘条件几何场预测”开展的开发轮次，包括但不限于：

- 文档补充
- 最小范围代码改动
- 评审与风险检查
- 每轮状态维护
- `project_memory` 长期记忆维护
- `handoff` 阶段性交接维护

本文档当前不服务于以下方向：

- Pointcept 通用文档维护
- 其他 backbone
- 其他数据集
- 复杂 loss 扩展
- 增强项探索
- 测试脚本扩展
- 预处理系统扩写

## 3. 执行期项目记忆与背景文档

当前执行期协作只依赖以下两类文档：

- 当前项目状态长期记忆：`semantic-boundary-field/project_memory/`
- 阶段性交接文档：`semantic-boundary-field/handoff/handoff_for_chat.md`

两者职责不同：

- `project_memory` 是当前唯一执行级项目记忆，用于记录当前已经成立、当前有效的项目状态、边界、规则和任务
- `handoff` 只服务阶段切换时的新 chat 快速接手，不参与日常执行约束

背景文档说明：

- `docs/research_plan_semantic_boundary_field.md` 只保留为项目历史背景、立项背景和归档参考
- `research_plan` 不参与当前轮执行约束

执行要求：

- 当前实现边界、当前状态、当前任务、当前规则，都以 `project_memory` 为准
- 若实现背景、当前结论、阶段状态、训练规则、loss 定义、数据定义与 `project_memory` 不一致，应先指出差异
- 不得把 `handoff` 当作长期记忆使用
- 不得把背景级 `research_plan` 当作当前执行判定依据

## 4. 角色定义

### 4.1 Developer

职责：

- 根据当前轮 prompt 实现最小必要改动
- 先确认最小影响文件集合，再开始修改
- 只处理本轮定义的单一清晰问题
- 保持实现与本项目当前主线及 Pointcept 接口兼容

新增职责：

- 在开始当前轮实现前，优先读取与本轮直接相关的 `project_memory` 文件
- 以 `project_memory` 中的当前有效状态作为实现背景
- 若发现 `project_memory` 与当前代码、日志结论或已确认状态不一致，应明确指出，而不是直接忽略

职责边界：

- 可以做最小范围阅读
- 可以修改当前轮明确允许修改的文件
- 不得擅自扩大为架构重写
- 不得在未说明原因时扩展阅读范围

Developer 当前应优先遵守的原则：

- 优先最小改动
- 优先复用现有主线
- 避免一次性大改
- 保持改动集中，方便回退

### 4.2 Reviewer

职责：

- 检查实现是否正确
- 检查改动是否过度侵入
- 检查是否违反当前范围约束
- 给出约束内的下一步建议

新增职责：

- 检查本轮改动是否与 `project_memory` 当前状态冲突
- 检查是否超出当前 `project_memory` 已定义边界
- 检查本轮完成后是否存在应同步到 `project_memory` 的状态变化但尚未维护

职责边界：

- Reviewer 的首要任务是找问题，不是扩展需求
- Reviewer 不应把评审轮变成新的设计轮
- Reviewer 不应脱离当前 `project_memory` 擅自扩展新方向

Reviewer 当前重点检查项：

- 是否只解决了一个清晰问题
- 是否先确认了最小影响文件集合
- 是否出现无理由的阅读范围扩张
- 是否引入了与当前阶段不匹配的复杂结构
- 是否与 `project_memory` 当前有效状态保持一致

### 4.3 Maintainer

职责：

- 在每轮结束后维护项目文档状态
- 维护当前范围、当前结论、待办和风险
- 确保仓库文档持续反映“现在做什么 / 现在不做什么”

新增职责：

- 明确维护两层文档：`project_memory` 长期记忆与 `handoff` 阶段性交接文档
- 在状态发生确定性变化时更新 `project_memory`
- 在阶段结束且准备切换网页 chat 主题时生成或刷新 `handoff` 文档

职责边界：

- Maintainer 只维护当前已确认状态，不擅自写入未达成共识的新方案
- Maintainer 不把文档更新变成大规模技术设计扩写
- Maintainer 不把 `handoff` 维护成实时长期记忆系统

Maintainer 需要维护的两层文档：

### 4.3.1 `project_memory`（长期）

当发生以下情况时，必须更新 `semantic-boundary-field/project_memory/`：

- 当前有效设计发生稳定变化
- 当前阶段切换
- loss 定义发生确定性变化
- architecture 发生确定性变化
- data definition 发生确定性变化
- training rules 发生确定性变化
- task queue 发生确定性变化
- 当前已确认结论替代旧结论

更新原则：

- 只写当前有效状态
- 不写讨论过程
- 不写未确认方案
- 不把废弃方案写入主结构
- 废弃结论仅在确有必要时进入归档位置，不回写为当前状态

### 4.3.2 `handoff` 文档（阶段性交接）

只有在以下时机才更新 `semantic-boundary-field/handoff/handoff_for_chat.md`：

- 当前任务结束
- 准备新开网页 GPT chat 窗口
- 准备切换到新的独立讨论主题

维护原则：

- `handoff` 不是长期记忆
- `handoff` 不是 `project_memory` 的缩写版
- `handoff` 只服务“新 chat 快速接手”
- `handoff` 只保留下一窗口讨论最需要的最小背景

## 5. 每轮工作流程

每一轮工作按以下顺序执行：

1. 先确认本轮只解决一个清晰问题。
2. 先读取与本轮直接相关的 `project_memory` 文件。
3. 先确认当前有效状态和当前边界。
4. 再确认最小影响文件集合。
5. 若当前文件集合不足以完成任务，必须先说明为什么要扩大阅读范围。
6. 若发现 `project_memory` 与当前代码、日志或结论不一致，必须先指出差异。
7. 在范围被明确后，再执行最小必要修改。
8. 完成后由 Reviewer 检查正确性、侵入性、可维护性、越界风险以及与 `project_memory` 的一致性。
9. 由 Maintainer 判断本轮结果是否需要同步到 `project_memory`。
10. 若当前阶段结束且准备切换网页 chat 讨论主题，再生成或刷新 `handoff` 文档。

禁止跳步：

- 不先定范围就开始改动
- 不先确认文件集合就开始扩展实现
- 不先读取相关 `project_memory` 就直接进入实现
- 不在同一轮同时推进多个大问题
- 不把 `handoff` 当作每轮都必须实时维护的长期文档

## 6. 范围控制规则

所有 Agent 都必须遵守以下规则：

- 每轮只解决一个清晰问题
- 改动前先确认最小影响文件集合
- 若需扩大阅读范围，必须先说明原因
- 优先最小改动，避免一次性大改
- 优先复用本项目既有主线与 Pointcept 上游接口
- 不得因为“顺手”而追加无关改动
- 不得把 Pointcept 当作主项目来维护协作文档

当前范围控制的具体含义：

- 当前主项目是 `semantic-boundary-field`
- Pointcept 只作为上游依赖和兼容接口来源
- 当前策略只允许最小侵入式推进
- 未进入的 Phase 不应被提前实现

## 7. 输出格式要求

后续各轮输出应尽量保持低歧义、可审阅、可维护。

最低要求：

- 明确本轮目标
- 明确本轮实际改动文件
- 明确为什么这些文件足够
- 明确当前没有做什么
- 明确风险与下一步最小建议

若本轮包含文档修改：

- 应给出最终落盘文件路径
- 应给出完整正文或足够完整的结构化摘要

若本轮涉及 `project_memory` 判断：

- 应说明读取了哪些相关记忆文件
- 应说明本轮是否触发 `project_memory` 更新
- 若未更新但存在差异，应明确指出原因

若本轮涉及阶段切换：

- 应说明是否需要生成或刷新 `handoff`

若本轮包含代码修改：

- 应列出改动文件
- 应说明改动目的
- 应说明潜在风险
- 应说明是否运行验证以及未验证部分

## 8. 禁止事项

当前阶段明确禁止：

- 忽视与本轮直接相关的 `project_memory`
- 脱离当前 `project_memory` 自行扩展任务目标
- 未经说明就扩大阅读范围
- 一次性大改 backbone、训练主线或配置体系
- 擅自切换到其他 backbone 或其他数据集
- 提前引入复杂 loss、增强项或额外局部图模块
- 把当前轮变成多目标并行推进
- 在未被要求时修改无关 Python 代码或配置文件
- 把文档维护轮变成代码实现轮
- 把 `project_memory` 写成聊天记录归档
- 把 `handoff` 写成实时维护系统
- 把 Pointcept 通用维护事项混入本项目协作文档主结构

## 9. 文档维护规则

本文档必须持续维护，但维护动作也必须受控。

维护原则：

- 只记录当前已经确认的规则
- 不记录未经执行或未经确认的假设
- 每次更新都应服务于降低歧义和降低范围失控风险
- 本文档本身也属于需要维护的协作文档
- 本文档当前服务的是执行期协作机制
- 执行期协作机制只依赖 `project_memory` 与 `handoff`
- `research_plan` 只保留为历史背景参考，不属于执行期协作文档主结构

允许更新本文档的条件：

- 当项目协作流程发生结构性变化
- 当引入新的长期记忆或交接机制
- 当现有角色职责已不能覆盖当前工作流

更新边界：

- 可以补充协作规则、记忆维护规则、流程控制规则
- 不应把本文档扩写成技术方案书
- 不应把当前轮临时习惯直接固化为长期规则

Maintainer 在每轮结束后应检查：

- 本文档是否仍与 `semantic-boundary-field/project_memory/` 的职责边界一致
- 当前范围描述是否过时
- 当前禁止事项是否仍清晰
- 当前工作流程是否仍可直接执行

## 10. 当前执行结论

在当前项目阶段，所有 Agent 都应遵守以下统一结论：

- 现在做的是 `semantic-boundary-field` 的受控多阶段开发，不是开放式探索
- 现在优先做最小改动，不追求一步到位
- 现在优先维护本项目状态、范围和接口约束，不扩散为 Pointcept 通用维护
- 现在只以 `project_memory` 作为执行级项目状态来源
- 现在只在阶段切换或准备开新 chat 时维护 `handoff`
- 现在不再以 `research_plan` 约束执行轮次
- `research_plan` 仅作为背景参考，不参与当前开发判定

简化判断标准：

- 如果某个动作会扩大范围，就先停下来说明原因
- 如果某个动作不是本轮唯一问题所必需，就先不要做
- 如果某个新增想法没有被当前 `project_memory` 支持，就不要在当前轮推进
- 如果某个结论已经成为当前有效状态，就判断是否需要同步到 `project_memory`
- 如果只是为了下一窗口快速接手，才考虑更新 `handoff`
