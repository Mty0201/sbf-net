# AGENTS.md

## 作用范围

- 本文件只补充 `data_pre/bf_edge_v3/` 子目录的本地约束。
- 全局协作协议以项目根目录 `semantic-boundary-field/AGENTS.md` 为准。
- 角色职责不再在本文件中模拟；统一由 `semantic-boundary-field/.codex/agents/` 与 `semantic-boundary-field/.codex/skills/` 承担。

## 本地目标

当前重构目标是：在保持现有场景输入格式兼容的前提下，把 `bf_edge_v2` 逐步收敛为一个面向逐点边界监督生成的稳定 pipeline。

输入保持兼容：

- `coord.npy`
- `segment.npy`
- 可选 `normal.npy`
- 可选 `color.npy`

端到端核心输出应当始终围绕逐点边界监督：

- `edge_dist`
- `edge_dir`
- 必要时的 `edge_mask`
- 必要时的 `edge_support_id`
- 必要时的调试或辅助索引信息

中间表示包括但不限于：

- `boundary_candidates`
- `boundary_centers`
- `local_clusters`
- `supports`

这些都只是中间产物，不是最终目标本身。所有设计、实现、重构和验收，都必须优先服务于：

**逐点距离场和方向场的质量、稳定性、连续性、可监督性。**

不要让中间几何表示反过来绑架整体设计。


## 本地执行方式

- 仍然遵守“小步改动、兼容优先、结构清晰、验收优先”的原则。
- 本地工作如果会影响正式训练字段、`edge.npy` 语义或 `project_memory/03_data_pipeline.md`，必须同步检查主项目事实文档。
- 这里只保留预处理子目录自己的边界，不再在本文件内要求固定多角色输出格式。


## 工作原则

### 1. 小步重构原则

- 每轮只做一个小任务
- 优先做可回滚、可对比、可验收的改动
- 禁止一轮内大面积重写多个核心脚本
- 如果一个任务需要同时重写多个主流程模块，应先继续拆小

### 2. 目标对齐原则

- 不要为了中间几何表达而优化中间几何表达本身
- 始终围绕最终 pointwise vector field 输出思考
- `clustering / splitting / fitting` 的存在意义，必须回到最终边界监督质量
- 对局部结构的判断标准，不能只看“几何上是否好看”，还要看是否真正提升最终逐点场质量

### 3. 兼容性原则

- 输入格式必须保持兼容
- 非必要不破坏现有 CLI / 入口
- 非必要不修改已有输出文件命名
- 非必要不修改已有目录约定
- 如果必须调整，必须明确说明影响范围、兼容性风险和迁移方式

### 4. 结构优先原则

- 优先拆职责，再改算法
- 优先澄清 pipeline，再做性能或效果优化
- 优先消除多职责脚本、长函数、隐式依赖
- 在结构边界不清楚之前，不要急于替换算法或堆更多规则

### 5. 验收优先原则

- 每轮完成后必须先判断“这一步是否真的完成”
- 验收结论必须具体，不能只说“已完成”
- 必须指出风险、边界情况和未覆盖内容
- 验收必须回到本轮目标，而不是泛泛而谈


## 当前项目的重构重点

当前重点不是重写全部 pipeline，也不是立刻替换所有算法。

当前重点是：**逐步把现有流程转成“面向最终 vector field 输出”的结构。**

在当前仓库里，最需要优先关注的是 `boundary centers` 之后的这段链路：

- clustering
- refinement
- splitting
- merge
- post-RANSAC split

原因：

- 这里是当前复杂度最高的部分
- 这里最容易混入“为 support 拟合服务”的历史逻辑
- 这里也是最可能影响最终 `edge_dist / edge_dir` 质量的瓶颈区域

工作时要持续警惕：

- 不要被 supports 导向思维绑架整体设计
- 不要默认 cluster 的目标是“更好拟合 line/polyline”
- 要让中间模块更独立、边界更清晰、便于单独调试和验收
- 要逐步把评价标准从“几何拟合友好”转向“监督输出友好”


## 推荐任务优先级

这是长期优先级，不表示一轮内完成。

### Priority 1

- 明确端到端主流程与模块边界
- 降低脚本多职责问题
- 固化稳定的输入输出接口
- 明确哪些中间结果是主流程必需，哪些只是 debug 产物

### Priority 2

- 拆分 `clustering / splitting / refinement`
- 让这些模块可独立运行、单独验收
- 显式化中间数据结构和依赖关系
- 降低 `boundary_centers -> local_clusters -> supports -> vector_field` 链路中的隐式耦合

### Priority 3

- 建立面向 vector field 质量的验收标准
- 明确什么样的 `cluster / support / representation` 才真正有助于最终监督
- 让评估不只围绕 residual、linearity、拟合外观展开

### Priority 4

- 再考虑算法增强、参数改进、替换局部策略
- 在结构边界清晰后再比较 DBSCAN、细分规则、拟合策略的优劣
- 避免在结构混乱时做局部参数优化


## 禁止事项

在本仓库中，以下行为默认禁止：

- 不要未经说明直接大改目录结构
- 不要一轮同时修改多个主流程脚本并改变行为
- 不要为了“看起来更工程化”而空转重构
- 不要脱离最终监督目标去优化局部几何表示
- 不要只给建议不落地
- 不要只写代码不做验收
- 不要只做验收不规划下一轮
- 不要把一轮任务扩展成跨多个阶段的大重构
- 不要把“调参数”当成结构问题的替代方案
- 不要默认 supports 一定是最终必须保留的核心抽象


## 执行提醒

在这个仓库中，每一轮都应默认问自己四个问题：

1. 这一步是否真的直接或间接服务最终逐点边界场输出？
2. 这一步是否是一个足够小、可验收、可回滚的改动？
3. 这一步是否让结构更清楚，而不是更复杂？
4. 这一步完成后，是否能被明确验收、并在需要时同步回主项目文档？

如果这四个问题里有明显答不上来的项，就说明当前任务还不够好，应先继续缩小范围或重新定义本轮目标。
