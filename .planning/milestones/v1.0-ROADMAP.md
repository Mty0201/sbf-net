# Milestone v1.0: Workflow Migration

**Status:** Completed 2026-04-02
**Phases:** 1-4
**Total Plans:** 12

## Overview

This milestone migrated the repository from a hand-built workflow control layer to a GSD-first operating model. The shipped result preserves the minimum SBF-specific facts and training guardrails in canonical docs, archives legacy workflow surfaces for historical lookup, and makes `.planning/` the explicit control plane for future planning and execution.

## Phases

### Phase 1: Canonical SBF Guidance

**Goal:** Maintainers can find the minimum SBF-specific boundary rules, active architecture facts, experiment evidence, and training guardrails without relying on archived workflow scaffolding.
**Depends on:** Nothing (first phase)
**Plans:** 3 plans

Plans:

- [x] `01-01-PLAN.md` - Create the canonical facts/evidence source for repository boundary rules and current Stage-2 conclusions.
- [x] `01-02-PLAN.md` - Create the canonical training guardrails source for entrypoints, configs, and fail-fast runtime rules.
- [x] `01-03-PLAN.md` - Create the canonical guidance index and slim `AGENTS.md` plus the formal workflow doc to point to it.

**Details:**

- Maintainers can find SBF-vs-Pointcept boundary rules in a minimal canonical guide.
- Maintainers can confirm the current Stage-2 status and active `axis + side + support` mainline without opening archived workflow files.
- Maintainers can find the governing experiment evidence and training guardrails in canonical guidance.

### Phase 2: GSD Default Entry

**Goal:** Maintainers encounter GSD as the primary workflow system when they enter the repository through default-facing docs and thin wrappers.
**Depends on:** Phase 1
**Plans:** 3 plans

Plans:

- [x] `02-01-PLAN.md` - Rewrite root-facing repository docs so maintainers encounter GSD and canonical guidance first.
- [x] `02-02-PLAN.md` - Reframe `AGENTS.md` and the formal workflow doc as GSD-first control surfaces with preserved Stage-2 guardrails.
- [x] `02-03-PLAN.md` - Reduce retained thin wrappers and compatibility docs to explicit redirect-only legacy surfaces.

**Details:**

- Default-facing workflow docs identify GSD as the primary planning and execution system.
- Retained thin wrapper docs redirect maintainers to GSD instead of duplicating legacy orchestration logic.
- A maintainer following the repository's default workflow entry no longer gets sent to a parallel hand-built control path.

### Phase 3: Legacy Workflow Archival

**Goal:** Legacy orchestration scaffolding is archived or removed from the default path while useful historical knowledge remains discoverable.
**Depends on:** Phase 2
**Plans:** 4 plans

Plans:

- [x] `03-01-PLAN.md` - Create the archive framework and migration map before physical moves begin.
- [x] `03-02-PLAN.md` - Archive the handoff and project-memory continuity trees while preserving history.
- [x] `03-03-PLAN.md` - Archive legacy Codex-side orchestration tooling and record the retained GSD runtime subset.
- [x] `03-04-PLAN.md` - Archive remaining wrapper bodies and reduce live wrappers to redirect-only compatibility surfaces.

**Details:**

- `.codex/agents/`, `handoff/`, and `project_memory/` were archived or removed from the default workflow path.
- Repo-local skills, wrapper docs, hooks, and routing scripts used for manual orchestration were archived or reduced to non-default compatibility surfaces.
- Historical project knowledge remains discoverable under `docs/archive/workflow-legacy/`.

### Phase 4: Workflow Control Cutover

**Goal:** Maintainers can run future repository planning and execution through GSD and the `.planning/` control surface without using the legacy workflow layer as the control plane.
**Depends on:** Phase 3
**Plans:** 2 plans

Plans:

- [x] `04-01-PLAN.md` - Create the `.planning` operating guide and align the core planning artifacts to serve as the repository control plane.
- [x] `04-02-PLAN.md` - Route default-facing workflow docs and compatibility wrappers through the `.planning` control surface.

**Details:**

- Maintainers can start planning and execution from a GSD-first default entry without consulting the legacy workflow layer.
- Maintainers can move from entry surfaces to active planning artifacts under `.planning/` without relying on `handoff/` or `project_memory` as the control plane.
- Future workflow planning for this repository can proceed through GSD artifacts and commands as the default operating path.

## Milestone Summary

**Key Accomplishments**

- Canonical SBF facts, experiment evidence, and training guardrails were consolidated under `docs/canonical/`.
- Root-facing docs and formal workflow references were rewritten so the repository opens as GSD-first.
- Legacy workflow material was archived under `docs/archive/workflow-legacy/` instead of remaining in the default path.
- `.planning/README.md` was introduced as the explicit repo-local operating guide for future planning and execution.
- Compatibility wrappers were reduced to redirect-only surfaces.

**Key Decisions**

- GSD plus `.planning/` is now the active workflow control path for this repository.
- Canonical repository-specific facts stay in `docs/canonical/`, separate from workflow-control material.
- Legacy workflow material is preserved for lookup but is no longer a default operating surface.
- Thin wrappers are allowed only as compatibility redirects.

**Issues Resolved**

- Eliminated the repo's parallel default control path spread across root docs, wrappers, `handoff/`, and `project_memory/`.
- Preserved historical continuity material without letting it remain the active workflow layer.
- Centralized default entry, state, roadmap, and phase artifacts under `.planning/`.

**Issues Deferred**

- Full Nyquist validation artifact coverage for Phases 1-4 was not added during this milestone.

**Technical Debt**

- `docs/canonical/sbf_facts.md` still has minor provenance drift in three evidence rows that reference `AGENTS.md`.
- Milestone validation documentation is incomplete because no phase has a `*-VALIDATION.md` file.
