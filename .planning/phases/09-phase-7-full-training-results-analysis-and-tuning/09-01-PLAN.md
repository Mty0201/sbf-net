Create `scripts/analysis/parse_train_log.py` with the following behavior.

**CLI interface:**
```
python scripts/analysis/parse_train_log.py <train_log_path> [--output-dir DIR]
```
- `train_log_path`: path to a train.log file
- `--output-dir`: optional override for CSV output directory (defaults to same directory as the train.log)

**Auto-detection of run type:**
The script must auto-detect the run type by scanning the first 200 lines for one of these patterns:
- `"loss_focus="` in a `Train result:` line --> type = `focus` (active route, SupportGuidedSemanticFocusLoss)
- `"loss_edge="` in a `Train result:` line --> type = `support_only` (SemanticBoundaryLoss baseline)
- If neither found in first 200 lines, scan up to 5000 lines. If still not found, exit with error.

**Line-by-line parsing (per D-05):**
The script MUST NOT read the full file into memory. Use standard Python file iteration (`for line in open(...)`) which reads line-by-line.

**Extraction targets — per eval epoch:**

For type `focus` (active route), produce `metrics_epoch.csv` with columns:
```
epoch,val_mIoU,val_mAcc,val_allAcc,val_boundary_mIoU,val_boundary_mAcc,boundary_point_ratio,support_bce,support_cover,valid_ratio,train_loss,train_loss_semantic,train_loss_support,train_loss_focus
```

For type `support_only`, produce `metrics_epoch.csv` with columns:
```
epoch,val_mIoU,val_mAcc,val_allAcc,train_loss,train_loss_semantic,train_loss_edge,train_loss_support,train_loss_support_reg,train_loss_support_cover,train_loss_dir,train_loss_dist
```

**Parse patterns (derived from trainer.py):**

1. **Train result line** — matches `"Train result: "` prefix:
   - Focus type: extract `loss=`, `loss_semantic=`, `loss_support=`, `loss_focus=` using regex `r'(\w+)=([\d.]+)'`
   - Support-only type: extract `loss=`, `loss_semantic=`, `loss_edge=`, `loss_support=`, `loss_support_reg=`, `loss_support_cover=`, `loss_dir=`, `loss_dist=` using same regex

2. **Val result summary line** — matches `"Val result: mIoU/mAcc/allAcc "`:
   - Extract three floats from pattern `r'Val result: mIoU/mAcc/allAcc ([\d.]+)/([\d.]+)/([\d.]+)'`

3. **Per-batch validation line** (focus type only) — matches `"Val/Test: ["`:
   - For the LAST batch per epoch (highest iter), extract `val_boundary_mIoU`, `val_boundary_mAcc`, `boundary_point_ratio`, `support_bce`, `support_cover`, `valid_ratio`
   - Use regex `r'val_boundary_mIoU: ([\d.]+)'` etc.
   - Note: the trainer logs running averages per batch. The final batch value is NOT the epoch average. The epoch average comes from `metric_meters[key].avg`, which is what the Val result summary uses for mIoU/mAcc/allAcc. For boundary metrics, there is no separate summary line, so use the last batch's running average (which equals the epoch average since AverageMeter tracks cumulative avg).

4. **Per-class result line** — matches `"{class_name} Result: iou/accuracy "`:
   - Extract class name and iou/accuracy from `r'(\w+) Result: iou/accuracy ([\d.]+)/([\d.]+)'`
   - Class names: balustrade, balcony, advboard, wall, eave, column, window, clutter
   - Produce `per_class_iou.csv` with columns: `epoch,balustrade,balcony,advboard,wall,eave,column,window,clutter`

5. **Epoch counter** — The log does NOT print an explicit "Epoch N" header before validation. Instead, count occurrences of `"Val result:"` lines to derive epoch number. The eval_epoch config is 100, so epoch 1 in the CSV = the first eval (after 100 training epochs), epoch 2 = second eval (after 200 training epochs), etc. Add a `training_epoch` column that is `epoch * eval_epoch_interval` where `eval_epoch_interval` is derived: for active route (100 val results across 2000 total epochs) = 20 eval epochs means interval is 100. Actually, just use sequential epoch numbering 1..N for simplicity, and note the eval_epoch interval in the CSV header comment.

   Correction: active route has 100 val results (2000 epochs / eval_epoch=100 * train_loop = 2000/100 = 20 displayed epochs, so 20 val results). Wait, let me reconsider. The trainer config says `total_epoch=2000, eval_epoch=100`, so `train_loop = 2000/100 = 20` and `max_epoch = 100`. It runs `for epoch in range(1, max_epoch+1)` = 100 epochs displayed, each consisting of 20 loops over the dataset. So there are 100 eval epochs. **Epoch tracking logic:**
- Maintain a counter `eval_epoch = 0`
- When a `Train result:` line is found, store the train metrics for the current epoch
- When a `Val result:` line is found, increment `eval_epoch`, pair with stored train metrics and any boundary metrics collected since last Val result
- When per-class result lines are found (8 consecutive lines after Val result), store them keyed by current eval_epoch

**Output files:**
- `{output_dir}/metrics_epoch.csv` — one row per eval epoch
- `{output_dir}/per_class_iou.csv` — one row per eval epoch with per-class IoU
- Print summary to stdout: run type detected, number of eval epochs found, best val_mIoU and its epoch, output file paths

**Error handling:**
- If train.log does not exist, exit with clear error message
- If zero eval epochs found, exit with error
- Use `argparse` for CLI
